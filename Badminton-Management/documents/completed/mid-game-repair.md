# Mid-Game Re-pairing (จับคู่ใหม่กลางคัน)

## Feature specification
เพิ่ม logic การจับคู่ใหม่ "กลางคัน" สำหรับกรณีที่รายชื่อผู้เล่นเปลี่ยนระหว่างการแข่งขัน
(มีคนออก หรือมีคนมาใหม่) ทั้งที่ยังเล่นไม่ครบทุกคู่ที่ระบบจับไว้

ปัจจุบันปุ่ม **จับคู่** / **สุ่มคู่ใหม่** จะลบ `state.matches` ทิ้งทั้งหมดแล้วสร้างใหม่
ตั้งแต่ต้น — คู่ที่เล่นไปแล้ว (กรอกผลแล้ว) ในตารางจะหายไป (แม้คะแนนสะสมใน
`state.scores` ยังอยู่)

ฟีเจอร์นี้เพิ่มทางเลือกใหม่: **เก็บคู่ที่เล่นไปแล้ว ตัดเฉพาะคู่ที่ยังไม่ได้เล่นทิ้ง
แล้วจับคู่ผู้เล่นปัจจุบันต่อ** เพื่อให้เกมดำเนินต่อได้หลังเปลี่ยนรายชื่อกลางคัน

รองรับทั้งสองทิศทางด้วยปุ่มเดียวกัน (ปรับรายชื่อก่อน แล้วกดปุ่ม):
- **คนออกกลางคัน** — ลบชื่อออกก่อน → คนนั้นค้างสถิติไว้แค่ที่เล่นจริง คนที่เหลือเล่นต่อ
- **คนมาใหม่กลางคัน** — เพิ่มชื่อก่อน → คนใหม่เริ่มที่ 0 เกม จึงถูกจับคู่บ่อยจนไล่ทันคนอื่น

**กฎปัจจุบัน (ใช้สนามครบเป็นหลัก):** ทุกรอบเปิดสนาม **ครบทุกสนามเสมอ** (คนเล่นน้อยสุดลงก่อน —
ถ้าคนมากกว่าที่นั่ง คนที่นำคือคนที่ได้พัก) แล้วหยุดเมื่อ **คนที่ตามหลังไล่ทันระดับของคนที่นำสุด
ตอนเริ่มจับคู่ต่อ** (`min(playCount) >= initialMax`) สนามจะถูกเว้นว่าง **เฉพาะเมื่อคนไม่พอ 4 คน
ต่อสนามเท่านั้น** — ไม่เว้นเพื่อรักษาความเท่ากันอีกต่อไป จำนวนเกมจึงอาจต่างกันได้ (โดยทั่วไป ±1;
ในเคสที่คนพอดีเต็มสนามแล้วมีช่องว่างมากกว่า 1 อาจต่างเกิน ±1) เพราะให้น้ำหนัก "ใช้สนามครบ" เหนือ
"เท่ากันเป๊ะ"

ตัวอย่าง 8 คน/2 สนาม (พอดีเต็มสนาม ไม่มีใครได้พัก) ที่ 4 คนนำอยู่ 1 เกม → รอบไล่ทันเปิด **ทั้ง 2
สนาม**: 4 คนที่ตามไล่ทัน ส่วน 4 คนที่นำได้เล่นต่อ จบที่ห่างกัน 1 เกม (ไม่เว้นสนามให้ว่าง)

> หมายเหตุประวัติการตัดสินใจ:
> 1. v1 = "catch-up หยุดที่ระดับสูงสุด" — ถูกมองว่าดึงคนที่นำไปเล่นเกิน จึงเปลี่ยน
> 2. v2 = "เล่นต่อจนเท่ากันเป๊ะ (min===max)" — แต่ทำให้เกิดสนามว่างในรอบไล่ทัน (เช่น 8 คน/2 สนาม
>    รอบจบเปิดแค่ 1 สนาม) ซึ่งเสียดายเวลาเล่น
> 3. v3 (ปัจจุบัน) = ผู้ใช้ยืนยันว่า **ใช้สนามครบสำคัญกว่า ยอมให้จำนวนเกมต่างกันได้** จึงกลับไป
>    ใช้ catch-up หยุดที่ `min >= initialMax` แต่ **ไม่เว้นสนามว่างเพื่อความเท่ากัน** อีก

## User stories
- **As an** ผู้จัดการแข่ง **I want** เอาผู้เล่นที่ออกไปกลางคันออกจากรายชื่อแล้วกดจับคู่ต่อ
  **so that** เกมดำเนินต่อโดยไม่ต้องเริ่มใหม่และไม่เสียผลที่เล่นไปแล้ว
- **As an** ผู้จัดการแข่ง **I want** ผู้เล่นที่เหลือได้แข่งจำนวนเท่ากัน
  **so that** ยังยุติธรรมเหมือน logic เดิม
- **As an** ผู้จัดการแข่ง **I want** ผู้ที่ออกไปแล้วค้างสถิติไว้แค่ที่เล่นจริง
  **so that** กระดานคะแนนสะท้อนความจริง
- **As an** ผู้จัดการแข่ง **I want** เพิ่มผู้เล่นที่มาสายแล้วกดจับคู่ต่อ
  **so that** คนมาใหม่ได้ลงเล่นและถูกดึงขึ้นมาให้ทันคนอื่น

## Acceptance criteria
1. มีปุ่มแยก **"↻ จับคู่ต่อ (กลางคัน)"** บนหน้าตาราง แสดงเมื่อมีตารางอยู่
   (`state.matches.length > 0`)
2. กดปุ่มแล้ว:
   - **เก็บ** ทุกคู่ที่กรอกผลแล้ว (`scoreA != null && scoreB != null`) ไว้ครบ
   - **ตัด** ทุกคู่ที่ยังไม่ได้กรอกผล (คู่อนาคต) ทิ้ง
   - **จับคู่ใหม่** เฉพาะผู้เล่นที่อยู่ใน `state.players` ปัจจุบัน (คนที่ถูกลบจะไม่ถูกจับคู่อีก
     ส่วนคนที่เพิ่มเข้ามาใหม่จะถูกนำเข้าจับคู่) ต่อท้ายคู่ที่เก็บไว้
3. logic: นับเกมที่เล่นไปแล้วต่อเนื่อง (ไม่รีเซ็ตเป็น 0) แล้วจับคู่ต่อไปเรื่อยๆ (fewest-first
   **เปิดสนามครบทุกรอบ**) **จนคนที่ตามหลังไล่ทันระดับคนที่นำสุดตอนเริ่ม** (`min >= initialMax`)
   แล้วหยุด — จำนวนเกมอาจต่างกันได้ (ให้น้ำหนักใช้สนามครบเหนือเท่ากันเป๊ะ)
4. ยังคงเลือกคู่ขาที่ไม่เคยจับคู่กันมาก่อน (partner freshness) ต่อเนื่องจากคู่ที่เล่นไปแล้ว
5. ผู้ที่ออกไปแล้วยังปรากฏในคู่ที่เล่นไปแล้ว + สถิติสะสมเดิม (แข่งได้แค่ไหนก็แค่นั้น)
6. คะแนนสะสม (`state.scores`) ไม่ถูกแตะ — เพราะตัดเฉพาะคู่ที่ยังไม่มีผล

## Scope & out of scope
**In scope:** doubles + singles, ปุ่มใหม่, ฟังก์ชัน `continueSchedule` ใน schedule.js, เทสต์
**Out of scope:** UI เลือกเจาะจงว่า "ใครออก" (ใช้การลบรายชื่อเดิม), การย้อน/undo การจับคู่ต่อ,
การแจ้งเตือนข้ามอุปกรณ์

## Architecture & technical design
- **schedule.js**: เพิ่ม pure function `continueSchedule({ players, mode, courts,
  playedMatches, getPlayers, rand })` → คืน array ของคู่ใหม่ (round ต่อจากคู่ที่เล่นแล้ว)
  - นับ `playCount` ของผู้เล่นปัจจุบันจาก `playedMatches`
  - นับ `partnerCount` (doubles) จาก `playedMatches` เพื่อคงความสดของคู่ขา
  - **วนจนไล่ทัน:** แต่ละรอบ sort fewest-first (playCount แล้ว lastPlayed)
    คนที่เล่นมากสุดจึงถูก sort ไปท้ายและได้พัก (เฉพาะเมื่อคนมากกว่าที่นั่ง)
  - **ใช้สนามครบเสมอ:** แต่ละรอบเปิดคอร์ตครบตาม `courts` (fewest-first) — เว้นสนามเฉพาะเมื่อ
    คนเหลือไม่พอ 4 คนต่อสนาม ไม่มีการเช็ก spread เพื่อหยุดเปิดคอร์ตอีกแล้ว
  - **เงื่อนไขหยุด:** เช็ก `min(playCount) >= initialMax` ก่อนเริ่มแต่ละรอบ (initialMax = เกมสูงสุด
    ตอนเริ่มจับคู่ต่อ) → หยุดเมื่อคนที่ตามหลังไล่ถึงระดับคนที่นำสุด
  - **รับประกันหยุด:** ทุกรอบจะมีคนที่ตามอยู่อย่างน้อย 4 คนได้เล่น (+1) เสมอ `min` จึงไต่ขึ้นเข้าหา
    `initialMax` เรื่อยๆ มี ceiling +1000 เป็น backstop
  - ถ้าไม่มีใครเคยเล่นเลย (max playCount = 0) → fallback ไปเรียก `makeSchedule` ปกติ (สร้างเต็ม)
- **script.js**: `continueScheduleMidGame()` — กรองคู่ที่เล่นแล้ว, เรียก `continueSchedule`,
  set `state.matches = [...played, ...newMatches]`, save + render. ผูกปุ่มใหม่
- **index.html / style.css**: ปุ่มใหม่ใน `.section-header-right` ใช้สไตล์เดียวกับ `.btn-reshuffle`

## Edge cases & error handling
- ยังไม่ได้เล่นคู่ไหนเลย → fallback สร้างตารางเต็ม (ไม่ลบทิ้งเปล่า)
- ผู้เล่นปัจจุบันเท่ากันหมดอยู่แล้ว → ไม่มีคู่เพิ่ม (caller แจ้ง "เท่ากันอยู่แล้ว")
- ผู้เล่นน้อยกว่าจำนวนต่อคอร์ต (doubles<4, singles<2) → สร้างคู่ไม่ได้, คืน []
- ความจุสนามมากพอจะนั่งทุกคนทุกตา (เช่น 8 คน/2 สนาม) → เปิดสนามครบทุกรอบ คนที่ตามไล่ทัน
  ส่วนคนที่นำเล่นต่อ จบที่ห่างกัน 1 เกม (ไม่เว้นสนามว่าง)
- เคสคนมาใหม่/คนออกที่ทำให้ผลรวมเกมหารไม่ลงตัว → ระบบเล่นต่อจนคนที่ตามไล่ถึง initialMax แล้วหยุด
  จำนวนเกมปลายทางอาจต่างกันได้ (ยอมรับได้ตามกฎ "ใช้สนามครบเป็นหลัก")

## Testing strategy
เพิ่มเทสต์ใน schedule.test.js:
- เก็บ playCount ต่อเนื่อง (ไม่รีเซ็ต), ผู้เล่นที่ออกไปไม่อยู่ในคู่ใหม่
- catch-up: laggard ไล่ถึง target, overshoot น้อยสุด, จบแบบ spread เล็ก
- **คนมาใหม่กลางคัน** ถูกดึงเข้ามาเล่นไล่ถึง target, การหยุดยังมีขอบเขต (bounded)
- partner freshness ต่อเนื่องจาก playedMatches
- target===0 → fallback เท่ากับ makeSchedule (ทุกคนได้เล่น)
- รับประกันหยุด (เทสต์รันจบ ไม่ค้าง 1000 รอบ)
- ผู้เล่นน้อยกว่าจำนวนต่อคอร์ต → คืน []
- **ความจุสนามมากพอจะนั่งทุกคน (8 คน/2 สนาม)** → เปิดสนามครบทั้ง 2 จบที่ห่างกัน 1 เกม
  (ครอบคลุมทั้งคนออกและคนเข้ากลางคันในสภาพนี้)
- **ใช้สนามครบทุกรอบ ไม่เว้นว่างเพื่อความเท่ากัน** → 9 คน/2 สนาม คนมาสาย: ทุกรอบใช้ครบ 2 สนาม
- regression: continueSchedule ไม่เว้นสนามว่างเพื่อปรับเท่ากัน (หลายสภาพ × หลาย seed)
- singles catch-up
